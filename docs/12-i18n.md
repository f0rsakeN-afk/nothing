# Internationalization (i18n)

## Overview

The app uses **next-intl** for internationalization, providing:
- Server and client-side translations
- Type-safe translation keys
- Pluralization support (ICU MessageFormat)
- RTL support for Arabic
- Cookie-based locale persistence

## Architecture

```
locales/
├── en.json    (English - source of truth)
├── es.json    (Spanish)
├── fr.json    (French)
├── de.json    (German)
├── zh.json    (Chinese)
├── ja.json    (Japanese)
└── ...

routing.ts        # Locale configuration
i18n.ts          # next-intl server config
middleware.ts     # Locale detection & cookie
```

## Usage in Components

### Basic Usage

```tsx
import { useTranslations } from "@/hooks/useTranslations";

function MyComponent() {
  const t = useTranslations();

  return (
    <Button>{t("common.save")}</Button>
    <p>{t("chat.newChat")}</p>
  );
}
```

### With Variables

```tsx
// {count} is replaced with the count value
t("chat.members", { count: 5 }) // "5 members"

// Pluralization works automatically
t("time.minutesAgo", { count: 3 }) // "3 minutes ago"
t("time.minutesAgo", { count: 1 }) // "1 minute ago"
```

### Namespaced

```tsx
// Use namespace parameter for cleaner code
const t = useTranslations("settings");
t("language") // "Language"
t("savedSuccessfully") // "Settings saved successfully"
```

### RTL Support

The layout automatically handles RTL for Arabic:

```tsx
// In app/layout.tsx
<html lang={locale} dir={locale === "ar" ? "rtl" : "ltr"}>
```

## Adding a New Language

1. **Create translation file:**
   ```bash
   cp messages/en.json messages/fr.json
   ```

2. **Translate strings** in `messages/fr.json`

3. **Add locale to routing.ts:**
   ```typescript
   export const routing = defineRouting({
     locales: [
       "en", "es", "fr", // Add new locale here
       // ...
     ],
   });
   ```

4. **Add locale metadata:**
   ```typescript
   export const SUPPORTED_LOCALES = [
     // ...
     { code: "fr", name: "French", nativeName: "Français", dir: "ltr", flag: "🇫🇷" },
   ];
   ```

## Server-Side Locale

```typescript
import { getLocaleFromCookie, getMessages } from "@/lib/i18n-utils";

// In Server Components
async function ServerPage() {
  const headersList = await headers();
  const cookieHeader = headersList.get("cookie");
  const locale = getLocaleFromCookie(cookieHeader);
  const messages = await getMessages(locale);

  return (
    <I18nProvider locale={locale} messages={messages}>
      <PageContent />
    </I18nProvider>
  );
}
```

## Type Safety

Translation keys are type-checked:

```typescript
// TypeScript will error if key doesn't exist
t("common.save")     // ✓ Works
t("common.invalid")  // ✗ TypeScript error
```

## Best Practices

1. **Use namespaces** - Keep translations organized by feature
2. **Use variables** - Don't concatenate strings: `t("chat.members", { count: n })`
3. **Handle pluralization** - Use ICU MessageFormat: `{count, plural, one {# item} other {# items}}`
4. **Default to English** - All new strings should be added to `en.json` first
5. **Don't translate AI responses** - AI chat responses are NOT translated (they respond in user's language naturally)

## Middleware

The middleware handles:
- Cookie reading for locale persistence
- Accept-Language header detection
- Setting default locale for new users

```typescript
// Cookie takes priority over Accept-Language
// 1. Check NEXT_LOCALE cookie
// 2. Fall back to Accept-Language header
// 3. Default to "en"
```
